# BAR 業務日報・売上管理システム 要件定義書

日々の売上報告・勤怠管理・目標進捗をスマートに一元管理し、手計算のミスをなくして
「正しい評価」と「利益の見える化」を行うための、日報・進捗管理システムの要件定義書
である。元仕様（Google ドキュメント）の内容を、実装フェーズにそのまま移行できる
粒度まで具体化する。

実装は `bar-dashboard/` ディレクトリ（React + TypeScript + Vite + Tailwind CSS）に
存在する。

## 1. 背景・目的

BAR事業（1部・2部の2部門、代表2名＋スタッフ複数名）において、日々の売上・経費・
勤怠報告は手作業の集計に依存しており、ミスや集計遅延が発生しやすい。本システムは
以下を目的とする。

- 日報（売上・経費・勤怠）の入力から、月次目標に対する進捗・インセンティブ・利益
  計算までを自動化する。
- 1部・2部それぞれの実績を、日次・月次・個人単位で可視化する。
- 「正しい評価」（インセンティブ・生産性の可視化）と「利益の見える化」（会社利益の
  可視化）を行う。

## 2. 運用フロー

1. 指定フォーマットの日報（売上・経費・勤怠報告）を LINE で送信する。
2. システムへデータが送信される。
3. 指定項目は追加できるようにする（フォーマットは固定しすぎない）。
4. システムにて、勤怠管理・売上実績管理・目標との進捗管理を実施する。
5. ダッシュボードで実績確認、月ごとのサマリー、日ごとの進捗状況を確認できる。
6. Bot への LINE 日報送信を起点に、管理システムへのデータ反映・集計を行い、集計
   結果から報告内容の最終版（★自動計算項目入り）を自動生成する。
7. 生成された日報を LINE へ転送し、報告完了とする。

### 2.1 本リポジトリでの実装範囲（スコープ外の明示）

LINE Bot 本体（Messaging API 連携）・サーバー側 DB・Google ドライブへの自動アップ
ロードは、認証情報や常設サーバーが必要なため本リポジトリの範囲外とする。本実装
では、LINE Bot からの送信を模した「日報入力フォーム」を用意し、送信すると仕様書
通りの★自動計算を行った「報告用」テキストを生成する。データは実運用の DB の代わり
にブラウザの `localStorage` に保存する（`invoice-tool/` と同様の方針）。伝票・
領収書アップロードはファイル名の記録のみ行い、実際の Google ドライブ保存は行わない。
将来的にサーバー API や LINE Messaging API Webhook を追加する場合も、本実装の
データモデル・計算ロジック（`src/lib/calculations.ts` 相当）はそのまま流用できる
構成とする。

## 3. データ定義

### 3.1 部門 (Department)

- `1部` / `2部` の2部門。代表・スタッフはいずれかの部門に所属する。

### 3.2 スタッフマスター (STAFF_MASTER)

| カラム名 | 論理名 | 説明 |
|---|---|---|
| id | スタッフID | ユニークID |
| name | 氏名 | 例：竹原、吉田、その他スタッフ |
| role | 役割 | `代表` \| `スタッフ` |
| department | 所属部門 | `1部` \| `2部` |
| hourlyWage | 時給 | 部門既定値で自動設定（1部 ¥1,300 / 2部 ¥1,600）。個別上書き可 |
| monthlyGoal | 個人月間目標売上 | スタッフ個人進捗管理で使用 |

初期データ：代表＝竹原（1部）、吉田（2部）。スタッフは各部門に複数名を仮登録する。

### 3.3 部門月間目標マスター (DEPARTMENT_GOAL)

| カラム名 | 説明 |
|---|---|
| department | `1部` \| `2部` |
| month | 対象月（YYYY-MM） |
| monthlySalesGoal | 当月目標売上（税込） |
| businessDays | 当月の営業日数（休業日を除いた日数） |

### 3.4 パラメータマスター (RATE_MASTER)

section 6（条件）を可変値としてマスター管理画面から編集可能にする。

| キー | 既定値 | 説明 |
|---|---|---|
| taxRate | 10% | 税抜換算に使用する消費税率 |
| hiaceIncentiveRate | 5% | ハイエース売上インセン（個人・代表） |
| hqIncentiveRate | 10% | 本部売上インセン |
| staffIncentiveRate | 10% | スタッフ売上インセン |
| hourlyWageDept1 | ¥1,300 | 1部勤怠時給 |
| hourlyWageDept2 | ¥1,600 | 2部勤怠時給 |

### 3.5 代表日報 (DAILY_REPORT_REP)

【入力】日付、部門（1部/2部）、報告者名、および以下。

1. **個人日報**：通常売上、ハイエース売上、既存来客数、新規来客数、クロスセル件数
2. **BAR全体日報**：現金売上、クレカ売上、電子マネー売上、QR決済売上、当日スタッフ
   売上（内訳）、当日イベント売上（内訳）、当日本部売上（内訳）、既存来客数、
   新規来客数
3. **総評・当日予定**：総評コメント、当日予定（出勤・予約・タスク）、伝票ファイル名
4. **経費**（複数行）：費目、金額、内容詳細、領収書ファイル名

同一日付・同一部門の代表日報が複数回送信された場合は、送信時刻 (`submittedAt`) が
最新のものを正として上書きする（3.7 節参照）。

### 3.6 スタッフ日報 (DAILY_REPORT_STAFF)

【入力】

1. **勤怠管理**：日付、報告者名、勤務部門（1部/2部）、勤務開始時刻、勤務終了時刻、
   休憩時間（分）、当日売上、売上詳細、既存来客数、新規来客数、クロスセル件数
2. **振り返り**：良い点、改善点

同一日付・同一スタッフの日報は、送信時刻が最新のもので上書きする。

### 3.7 データ修正ルール

> データ修正は、同一日付で報告時間が最新のもので上書きする。

→ 実装上は「日付＋報告者（＋部門）」をキーとして最新 `submittedAt` の1件のみを
有効な実績として扱う（Upsert）。過去の送信履歴は監査用に保持してよいが、集計には
最新分のみを使用する。

## 4. 報告用（自動生成）フォーマットと計算式

「★」は入力値ではなく、システムが集計・計算して自動生成する項目。

### 4.1 代表：個人日報

| 項目 | 算出式 |
|---|---|
| 個人当日総売上 ★ | 通常売上 + ハイエース売上 |
| ハイエース売上インセン ★ | ハイエース売上（税抜） × `hiaceIncentiveRate`(5%) |

### 4.2 代表：BAR全体日報

| 項目 | 算出式 |
|---|---|
| 全体当日総売上 ★ | 現金売上 + クレカ売上 + 電子マネー売上 + QR決済売上 |

当日スタッフ売上・当日イベント売上・当日本部売上は、上記合計の「内訳（財源別では
なく発生源別の内訳）」として入力されるため、整合チェック（3項目の合計が全体当日
総売上と一致するか）をダッシュボード上で警告表示する。

### 4.3 代表：BAR当月進捗状況（すべて★・部門単位で集計）

| 項目 | 算出式 |
|---|---|
| 当月目標 | DEPARTMENT_GOAL.monthlySalesGoal（当該部門・当該月） |
| 当月累計売上 | 当該部門・当該月の 全体当日総売上 の日次合計（月初〜報告日） |
| 残営業日数 | DEPARTMENT_GOAL.businessDays − 報告日までの経過営業日数 |
| 1日必達 | (当月目標 − 当月累計売上) ÷ 残営業日数（残営業日数が0以下の場合は0） |
| スタッフ売上累計（スタッフ別） | 当該月の各スタッフの「当日売上」の月間合計 |
| 本部売上累計 | 当該部門・当該月の「当日本部売上」の月間合計 |
| イベント売上累計 | 当該部門・当該月の「当日イベント売上」の月間合計 |

### 4.4 代表：経費（★項目）

| 項目 | 算出式 |
|---|---|
| 当月経費累計 | 当該部門・当該月の経費「金額」の月間合計 |
| 人件費累計 | 当該部門に属する全スタッフ・代表の (実働時間 × 時給) の月間合計 |
| ハイエース売上インセン累計 | 当該代表の当月のハイエース売上インセンの合計 |
| 本部売上インセン累計 | 本部売上累計（税抜） × `hqIncentiveRate`(10%) |

### 4.5 代表：利益・インセン（暫定・★）

| 項目 | 算出式 |
|---|---|
| 当月利益 | 当月累計売上（税抜） − 当月経費累計 − 人件費累計 − (ハイエース売上インセン累計 + 本部売上インセン累計 + スタッフ売上インセン累計) |
| 個人インセン | 当該代表の ハイエース売上インセン累計 + 本部売上インセン累計 |
| 会社利益 | 当月利益 − 個人インセン |

> 「利益は売上合計から小計で算出する（税抜）」の条件に従い、利益計算のみ税抜換算
> （金額 ÷ (1 + taxRate)）を用いる。それ以外の表示項目は税込のまま表示する。

### 4.6 スタッフ：当月進捗（★）

| 項目 | 算出式 |
|---|---|
| 当月目標 | STAFF_MASTER.monthlyGoal |
| 当月売上 | 当該スタッフの当月の「当日売上」の合計 |
| 時給 | 所属部門の `hourlyWageDept1/2`（個別上書きがあればそちらを優先） |
| スタッフ売上インセン | 当月売上（税抜） × `staffIncentiveRate`(10%) |
| 生産性（個人利益） | 当月売上（税抜） − 当月人件費（実働時間合計 × 時給） − スタッフ売上インセン |

> 「生産性（個人利益）」の算出式は元仕様に明記がないため、本要件定義書での実装上の
> 定義とする。運用開始後、実態に合わせて調整可能なようにロジックを1箇所
> （`src/lib/calculations.ts`）に集約する。

### 4.7 勤怠時間（実働時間）の計算

> 勤怠管理で、24時〜5時などの日またぎは、報告日付を正としシステムへ反映する。
> 23時〜5時勤務は6時間勤務。集計でマイナスにならないようにする。

- 勤務終了時刻が勤務開始時刻以下（日またぎ）の場合、終了時刻に24時間を加算してから
  差分を計算する（例：23:00〜5:00 → 30:00 − 23:00 = 7時間 … ただし休憩1時間なら
  6時間勤務）。
- 実働時間 = max(0, (終了時刻 − 開始時刻の分差、日またぎ補正後) − 休憩時間)。
  負値になる入力は 0 に丸める。
- 日またぎ勤務の売上・実績はすべて「報告日付」（＝出勤日）に計上する。

## 5. 画面要件

1. **日報入力**：代表用／スタッフ用のタブを切り替えて入力する画面。送信すると
   LINE Bot への送信を模し、★自動計算込みの「報告用」テキストを生成してプレビュー
   表示し、クリップボードにコピーできる（LINE グループへの転送を想定）。
2. **サマリー**：1部・2部それぞれの月次・年次の売上・経費・利益を一覧表示する。
3. **ダッシュボード**：1部・2部を切り替え、月ごとに以下を表示する。目標、売上、
   目標と売上のGAP、達成率、必要達成率、経費、固定費（人件費）、スタッフ時給、
   スタッフインセン、来客数、客単価、利益。
4. **日次進捗管理**：部門・月を切り替え、日ごとの売上・経費・利益、目標との
   GAP・達成率を一覧表示する。
5. **代表個人進捗管理**：代表・月を切り替え、月次の売上・利益・インセンの管理と
   日次の進捗を表示する。
6. **スタッフ個人進捗管理**：スタッフ・月を切り替え、月次の売上・勤怠・インセン・
   生産性の管理と日次の実績を表示する。
7. **マスター管理**：スタッフマスター、部門月間目標、パラメータ（インセン率・
   時給・税率）の追加・編集を行う。

## 6. 条件（計算パラメータの既定値）

- 利益は、売上合計から小計で算出する（税抜）。
- 利益計算以外では税込で数字を表示する。
- ハイエース売上インセン（個人）：ハイエース売上の5%。
- 本部売上インセン：本部売上の10%。
- 1部勤怠：時給1,300円。
- 2部勤怠：時給1,600円。
- スタッフ売上インセン：スタッフ売上の10%。
- インセンは全て売上の小計（税抜）で算出する。
- 勤怠管理の日またぎ処理は 4.7 節のロジックに従う。
- 伝票・領収書は本来 Google ドライブへ年・月・日フォルダで保存するが、本実装では
  ファイル名の記録のみとする（2.1節）。

## 7. 非機能要件・技術要件

- Frontend: React (SPA) + TypeScript + Vite + Tailwind CSS（`dashboard/` と同一
  スタック・同一デザイン方針）。
- データ永続化：ブラウザの `localStorage`（サーバーレス）。
- `alert()` / `confirm()` は使用せず、トースト通知またはモーダル UI を用いる
  （iframe / GitHub Pages 環境でも動作させるため）。
- エクスポート：報告用テキストのクリップボードコピー機能を標準搭載する。
