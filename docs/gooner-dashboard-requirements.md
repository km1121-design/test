# Gooner運送事業部 経営分析・黒字化ダッシュボードシステム要件定義書

Gooner運送事業部の財務改善、および「月利100万円の黒字化」の迅速な達成を目的とした、
インタラクティブ経営分析ダッシュボードシステムの要件定義書である。他開発エージェント
（AI）や開発担当会社が、ブレなく直ちに実装フェーズへ移行できるように必要な全仕様・
数式・データ構造をここに規定する。

実装は `dashboard/` ディレクトリ（React + TypeScript + Vite + Tailwind CSS）に存在する。

## 1. システム化の背景と目的

### 1.1 背景

Gooner運送事業部は、ラストマイル（ヤマト宅配受託）およびB2B企業配（スポット配送）を
展開しているが、現状月間約48万円の赤字を計上している。手元には豊富な現場データや
スプレッドシート、LINEの稼働・バグ報告ログが存在するものの、リアルタイムな経営PL、
セグメント別の損益ボトルネック、ドライバーごとの財務貢献度が経営陣に可視化されておらず、
迅速な意思決定が行えない状態にある。

### 1.2 目的

- **赤字構造の徹底的な可視化**: 全体PL、ヤマト宅配、企業配の3つの視点から、固定費・
  マージン率の課題を明確にする。
- **黒字化シミュレーターによる経営判断の迅速化**: 「あと何名ドライバーを増やせば
  黒字になるか？」等の What-If シミュレーションをリアルタイムに行い、最速で黒字化
  アクションを策定する。
- **現場摩擦（バグ・エラー）の可視化**: 現場のイライラ（離職リスク）を検知し、採用
  広告費（求人媒体費）の無駄な発生を抑止する。

## 2. 業務・機能システム範囲 (Scope)

本システムは以下の4つの分析ビュー（タブ）と、常時連動するリアルタイムシミュレーター
によって構成される。

```
+----------------------------------------------------------------------------------+
|                            経営ダッシュボードヘッダー                               |
+---------------------------------------+------------------------------------------+
|                                       |       右側：詳細分析ワークスペース             |
|                                       |  +-------------------------------------+ |
|  左側：What-If シミュレーター           |  | タブ選択: [全体PL] [ヤマト] [企業配] [現場] | |
|  - ヤマト稼働人数 (スライダー)         |  +-------------------------------------+ |
|  - 1人1日あたり配完数 (スライダー)      |  |                                     | |
|  - 企業配平均手数料率 (スライダー)     |  |   各タブに応じたグラフ・テーブル       | |
|  - 管理者人件費削減率 (スライダー)     |  |   分析レポ・経営陣への改善示唆         | |
|  - 車両関連固定費削減 (スライダー)     |  |                                     | |
|                                       |  +-------------------------------------+ |
+---------------------------------------+------------------------------------------+
```

## 3. データ要件・データマスター定義

システムで保持、または外部API/スプレッドシートから連携されるべきデータ構造。

### 3.1 ドライバーマスター (DRIVER_MASTER)

各ドライバーの基本契約、売上実績、会社から支払う外注費、発生した実費経費を管理する。

| カラム名 | 論理名 | データ型 | 説明 / 例 |
|---|---|---|---|
| id | ドライバーID | String | ユニークID。例: DRV-001 |
| name | 氏名 | String | ドライバー名 |
| role | 役割・ランク | String | オーナー, マネージャー, ディレクター, 一〜四等級, 研修生 |
| unitPrice | ドライバー単価 | Number | ドライバー受取単価。例: 155（会社受取単価との差額が粗利） |
| area | 担当エリア | String | 配送地域、または案件 |
| type | 雇用・委託形態 | String | 役員, フルコミ, 経費会社持ち, アルバイト |
| baseSalary | 固定人件費 | Number | 月額固定給（管理者用）。例: 赤羽 ¥400,000 |
| actualSales | 売上実績 | Number | 16日間（半月）時点での売上 |
| actualOutsource | 外注費支払実績 | Number | フルコミドライバー等への支払額 |
| actualExpense | 実費経費 | Number | 駐車場・ガソリン等、ドライバーごとの経費 |
| segment | 担当部門 | String | ヤマト, 企業配, 全体 |
| subSegment | サブ部門 | String | 宅配委託会社名。fate, k-dash |

### 3.2 固定諸経費マスタ (VEHICLE_EXPENSE_DETAIL)

車両関連および間接固定費を明確にし、黒字化のハードルを定義する。

| 項目名 | 金額 | 区分 | 説明 |

## 4. 計算ロジック・アルゴリズム要件

シミュレーターおよび各セグメントで実行される計算式を規定する。

### 4.1 ヤマト宅配部門のシミュレーションロジック

ヤマト宅配部門における月間シミュレーション売上・マージンの算出式：

- ヤマト売上 (Y_sales) = 稼働人数 (D) × 1日あたり配完数 (B) × 会社受取単価平均 (P_in) × 稼働日数 (T)
- ヤマト外注費 (Y_out) = 稼働人数 (D) × 1日あたり配完数 (B) × ドライバー支払単価平均 (P_out) × 稼働日数 (T)
- ヤマト部門粗利 (Y_margin) = Y_sales − Y_out

規定値:

- 稼働日数 (T) = 22日
- 会社受取単価平均 (P_in) = 167円/個
- ドライバー支払単価平均 (P_out) = 153円/個（150円〜160円の加重平均値）
- 1個あたりの会社マージン利益 = 14円

### 4.2 企業配部門のシミュレーションロジック

企業配の月間ベース売上（定常案件計 ¥1,500,000）を基礎とし、設定された交渉利益
（手数料）率を適用する。

- 企業配粗利 (E_profit) = 企業配売上ベース (1,500,000円) × 企業配交渉利益率 (%)

### 4.3 管理者（役員）人件費のシミュレーションロジック

管理者3名（赤羽、小川、三田）の合計固定人件費：

- 管理者人件費 (A_cost) = 850,000円 × (1 − 管理者人件費削減率 (%))

### 4.4 固定諸経費シミュレーションロジック

固定車両管理諸経費の算出式：

- 固定諸経費 (F_cost) = 771,038円 − 諸経費削減設定額 (C_reduction)

### 4.5 月間最終経常損益予測（最終KPI）

スライダー連動で随時再計算される最終的な利益式：

- 最終経常損益 (NetProfit) = (Y_sales − Y_out) + E_profit − A_cost − F_cost

## 5. 各タブ画面の詳細機能要件

### 5.1 全体PL分析画面 (Overall PL Tab)

- 財務サマリー（半月実績）: 売上額、外注支払原価、全体利益（現在赤字 ¥48.1万）、
  利益率（-15.88%）の表示。
- 構造的ボトルネックの分析提示:
  - ヤマトの低マージン（¥7〜¥12/個中抜き）が諸経費を賄えていない。
  - 車両・間接経費の過多（月¥77万の流出）。
  - 管理者人件費（¥85万）の重荷。
- 固定費内訳テーブル: VEHICLE_EXPENSE_DETAIL に基づく詳細金額と注記の表示。

### 5.2 ヤマト宅配分析画面 (Yamato Delivery Tab)

- KPIサマリー: 売上高、部門粗利、現在人員の表示。
- ドライバー個別貢献度一覧: DRIVER_MASTER から宅配担当（subSegmentに値がある者）を
  抽出し、メンバー別の売上、外注費、会社への利益貢献額（マージン）をグリッド表示する。
- ボトルネック解決策の明示: 1人あたりの配送完了数はすでに平均122個に達しており、
  「ヤマト正社員目標（100個）」を超えている優秀な状態。したがって、課題は「生産性
  向上」ではなく、「フルコミ人員の純増（採用）」である点（目標21名・あと10名増）を
  強調表示。

### 5.3 企業配・スポット分析画面 (Enterprise B2B Tab)

- KPIサマリー: 企業配・スポット実績、部門利益、担当人員の表示。
- 経費会社持ちドライバーの収支採算テーブル: 実売上から「ドライバー支払額」および
  「会社負担経費（ガソリン・駐車料金）」を差し引き、実質会社に残っている利益を
  可視化。
- 案件マスタの中抜き率比較: PROJECT_MASTER に基づく各案件の手数料率を表示し、
  「たいやき案件（33.3%）」「日比谷案件（19.35%・会社経費負担）」などの採算性を
  可視化。

### 5.4 現場課題・バグトラッカー画面 (Friction Log Tab)

- 現場イライラ検知一覧: FIELDBUGS_LOG に基づき、LINE Botや報告用Webフォームに
  寄せられている不満やシステムバグを「未対応」「保留」「修正済み」のステータスで
  表示。
- 経営リスクの警鐘: 報告フォームの重複エラーやシステム遅延はドライバーの離職に
  直結する。離職が起きた場合、再採用のために求人・広告ランニングコスト（Indeed、
  バイトル月合計 ¥133,000）が突発的に発生し、黒字化を妨げるボトルネックとなる
  ことを解説。

## 6. 非機能要件・技術要件

### 6.1 実装フレームワーク

- Frontend: React (Single Page Application)。
- Style: Tailwind CSS をベースとした、ダークモード基準のモダンなインダストリアル
  デザイン。

### 6.2 UX / UI要件

- インタラクティブ操作性: シミュレーターのスライダーを変更した際、即座に全体の
  進捗バー、利益予測値、ゴールGAPがタイムラグなし（`useMemo` によるキャッシュ活用）
  で再計算・再描画されること。
- エラー耐性 / CSP対応: 外部リソース（アイコン、画像等）を必要最低限にし、描画
  ブロックを防ぐ。アイコンは lucide-react 等をバンドルしてオフライン・iframe環境
  でも作動すること。ブラウザ依存の `alert()` や `confirm()` などの処理は iframe内
  で動作しない可能性があるため、一切使用せず、システム内部にトースト通知（Toast）
  またはモーダルUIを組み込む。

### 6.3 データ出力・ポータビリティ

シミュレーション結果（各変数値、予測売上、予測最終損益など）をクリップボードに
コピーしてLINEやスプレッドシートに貼り付けられる、エクスポート（テキスト整形出力）
機能を標準搭載すること。
