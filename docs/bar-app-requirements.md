# BAR 日報・経営分析アプリ 要件定義書 v1.0

日報の入力・集計・経営分析をひとつのアプリで完結させ、完成した日報と経営上必要な
情報を LINE へ**配信**するシステムの要件定義書である。旧方針（LINE Bot にテキストを
送信して集計する方式、`line-bot/` に実装）からの方針転換版であり、企画壁打ち
（論点A〜K）の決定事項を反映した確定版。

- 旧要件定義書: `docs/bar-dashboard-requirements.md`（データ項目・計算式の定義は
  本書からも参照する。計算ロジックは変更なし）
- 既存実装: `bar-dashboard/`（画面・計算エンジン）、`line-bot/`（スキーマ・計算
  エンジン・報告用テキスト生成）— いずれも新アプリの部品として流用する

## 1. 方針転換の要旨

| | BEFORE（LINE Bot集計型） | AFTER（本書・アプリ入力/LINE配信型） |
|---|---|---|
| 日報の入力 | LINEにテンプレテキストを送信 | アプリの入力フォーム（スマホ最適化） |
| 集計・分析 | Botが返信／画面はデモ | アプリ内で完結・実データに接続 |
| LINEの役割 | 入力・出力の両方 | **配信（受け取り）専用** |
| Webhook受信・テキストパース | 必要 | **廃止** |

## 2. 決定事項ログ（壁打ち論点A〜K）

| 論点 | 決定 |
|---|---|
| A. 入力の主体 | **スタッフも各自アプリで入力**（ロール別権限を実装） |
| B. LINE配信の範囲 | **フル装備**：①日報転送 ②日次サマリー ③未提出リマインド ④異常アラート（フェーズ分割） |
| C. 伝票・領収書 | **Googleドライブ連携**（元仕様通り、年/月/日フォルダ） |
| D. 認証 | **LINEログイン（LIFF）** |
| E. 締め時刻 | **部門ごとに設定可能**（部門マスターに持たせる） |
| F. スタッフ本人への公開範囲 | **売上＋インセンまで**（時給×時間・生産性は代表のみ） |
| G. 運用コスト | **できるだけ無料枠**（LINE無料枠200通/月に収める配信設計） |
| H. ドライブの保存先 | **会社のGoogle Workspace（共有ドライブ）**＋サービスアカウント書き込み |
| I. ホスティング | **Cloudflare無料枠へ載せ替え**（Pages + Workers + D1 + Cron Triggers） |
| J. 時刻の初期値 | **1部・2部とも締め＝翌日19:00、サマリー配信＝22:00** |
| K. スタッフ日報の転送 | **専用グループへ転送**。無料枠と両立させるため**締め後に1日1通のまとめ配信**（全件即時が必要になったらライトプランへ切替） |

## 3. 利用者と権限

| ロール | 人数 | ログイン | できること |
|---|---|---|---|
| 代表 | 2名 | LINEログイン | 代表日報の提出／全ダッシュボード閲覧／マスター管理／LINE配信設定 |
| スタッフ | 6名〜 | LINEログイン | 自分の日報の提出・修正／自分の「目標・売上・インセン・日別実績」の閲覧 |
| 閲覧者（任意） | — | LINEログイン | ダッシュボード閲覧のみ（本部・出資者向け。必要になった時点で追加） |

- スタッフ本人には**時給×実働時間・生産性（個人利益）・他人の数字は表示しない**（論点F）。
- 初回のみ「招待リンク」からLINEログインし、本人のLINEアカウント（userId）と
  スタッフマスターを紐付ける。招待フローに公式アカウントの友だち追加を組み込む
  （個人宛リマインドの前提）。以後はリッチメニューからワンタップで起動。

## 4. 機能要件

### 4.1 日報入力（Phase 1）

- 代表日報：個人売上／BAR全体売上／総評・当日予定／経費（複数行）。
  項目定義・★自動計算項目は旧要件定義書 4節の通り。
- スタッフ日報：勤怠（開始・終了・休憩）・当日売上・売上詳細・来客数・
  クロスセル・振り返り（良い点・改善点）。
- 前日コピー・下書き保存・提出前チェック（決済内訳と発生源内訳の不一致警告など）。
- 同一営業日・同一報告者の再提出は最新を正として上書き（旧3.7節ルール。履歴も保持）。
- 提出後に★自動計算込みの「報告用日報」プレビューを表示。

### 4.2 経営分析（Phase 1、グラフ類はPhase 3）

- 旧実装 `bar-dashboard/` の6画面（サマリー／ダッシュボード／日次進捗／代表個人
  進捗／スタッフ個人進捗／マスター管理）を実データ（D1）に接続して提供する。
- 計算式は旧要件定義書 4節・6節（インセン率・時給・税抜計算・日またぎ勤怠）を
  変更なしで踏襲。実装は `bar-dashboard/src/lib/calculations.ts`（＝
  `line-bot/src/lib/calculations.ts`）を移植する。
- Phase 3で追加：推移グラフ（売上・利益・来客数・客単価）、着地予測
  （現在ペースでの月末見込み）。

### 4.3 LINE配信

| # | 配信 | トリガー | 宛先 | 内容 | フェーズ |
|---|---|---|---|---|---|
| ① | 日報転送 | 代表日報の提出時（即時） | 日報グループ | 報告用日報（★自動計算込み） | 2 |
| ② | 日次サマリー | 毎日22:00 | 日報グループ | 前営業日実績（部門別売上・利益）／当月累計と達成率／残営業日数と1日必達／未提出者一覧 | 2 |
| ⑤ | スタッフ日報まとめ | 毎日22:00 | スタッフ日報グループ（専用） | 当営業日の全スタッフ日報を1通にまとめて転送 | 2 |
| ③ | 未提出リマインド | 毎日19:00（締め） | 未提出の本人 | 「昨日の日報が未提出です」＋日報画面リンク | 3 |
| ④ | 異常アラート | 日次判定 | 代表 | 達成ペース急落・経費超過（しきい値はマスター設定） | 3 |

**無料枠設計（論点G・K）**：LINE公式アカウント無料枠は月200通（グループpushは
1回＝1通）。試算：①約60通＋②約30通＋⑤約30通＋③約20〜40通＋④約5〜10通
＝**約145〜170通/月で無料枠内**。各配信はマスター管理画面からON/OFF可能とし、
当月の消費通数をアプリ内に表示、枠に近づいたら警告する。スタッフ日報の全件即時
転送が必要になったらライトプラン（月5,000通）へ切替。

### 4.4 1日の運用タイムライン

| 時刻 | 出来事 |
|---|---|
| 営業中〜閉店後 | 代表・スタッフが随時日報を提出。代表日報は即時①転送 |
| 翌日 19:00 | 前営業日分の締め。未提出者へ③リマインド |
| 翌日 22:00 | ②日次サマリー＋⑤スタッフ日報まとめを配信 |

- 締め時刻・配信時刻は部門マスターで変更可能（論点E・J）。
- 締め後の提出も受け付け、集計に即時反映する（締めはリマインドとサマリー確定の
  基準時刻であり、入力のロックではない）。
- 「営業日」の定義：日またぎ勤務（24時〜5時等）は報告日付＝出勤日に計上
  （旧要件定義書 4.7節と同一）。

### 4.5 伝票・領収書（Phase 2）

- 日報フォームから写真を撮影／選択してアップロード。
- 会社Workspaceの**共有ドライブ**へサービスアカウントで書き込み、
  `伝票/YYYY/MM/DD/` の年／月／日フォルダに自動整理。
- ファイル名は `日付_部門_報告者_種別NN.jpg`（例: `2026-07-03_1部_竹原_領収書01.jpg`）。
- 日報詳細画面からドライブ上のファイルへリンクする。

### 4.6 管理（Phase 1〜2）

- マスター管理：スタッフ（氏名・役割・部門・時給上書き・個人目標）、部門月間目標
  （目標売上・営業日数・締め時刻・配信時刻）、パラメータ（インセン率・時給・税率）。
- LINE紐付け管理：招待リンク発行・紐付け解除。
- LINE配信設定：配信先グループID・各配信のON/OFF・時刻。

## 5. システム構成（論点I）

```
LINE（リッチメニュー）
   │ タップ
   ▼
LIFFアプリ = フロントエンド（Cloudflare Pages）
   React + TypeScript + Tailwind ─ bar-dashboard を移植
   │ fetch（LIFF IDトークン付き）
   ▼
API（Cloudflare Workers / Hono）
   IDトークン検証・日報API・集計API・管理API
   │                     │
   ▼                     ▼
D1（SQLite互換）      Google Drive API（共有ドライブ・サービスアカウント）
   ▲
   │ Cron Triggers（毎日19:00 / 22:00 JST）
   └─ リマインド判定・サマリー生成 → LINE Messaging API（push）
```

- **すべてCloudflare無料枠内**：Pages（フロント）、Workers（API・100k req/日）、
  D1（DB・5GB）、Cron Triggers（定時ジョブ）。スリープなし・ディスク消失なし。
- D1はSQLite互換のため、`line-bot/src/db/schema.sql` のスキーマと
  計算エンジンはほぼそのまま移植できる。`better-sqlite3` / Express 依存部分は
  Hono + D1バインディングに書き換える。
- LINE連携は Messaging API（push・リッチメニュー）＋ LINEログイン（LIFF）。
  Webhook受信は不要（返信でなくpushのみ）。
- タイムゾーンは Asia/Tokyo 基準（Cron はUTC指定のため 19:00 JST = 10:00 UTC、
  22:00 JST = 13:00 UTC）。

## 7. 開発フェーズ

| フェーズ | 内容 | 完了時の状態 |
|---|---|---|
| **Phase 1** アプリ本体 | Cloudflare（Pages/Workers/D1）へ構築、LINEログイン・ロール別権限、日報入力（代表・スタッフ）・前日コピー、分析6画面のDB接続、マスター管理・紐付け管理 | 「日報＋分析」を業務投入できる |
| **Phase 2** 配信と証憑 | ①日報転送・②日次サマリー・⑤スタッフ日報まとめ、配信設定画面、写真アップロード＋共有ドライブ連携 | LINE配信込みの本運用 |
| **Phase 3** 運用強化 | ③未提出リマインド・④異常アラート、推移グラフ・着地予測 | 運用データを見て優先順位を調整 |

### 事前に必要な外部準備

1. **LINE Developers**：プロバイダー作成 → LINEログインチャネル（LIFF）と
   Messaging APIチャネルを作成。公式アカウントを日報グループ・スタッフ日報
   グループに招待し、リッチメニューを設定。
2. **Google Cloud**：サービスアカウント作成 → Drive API有効化 → 会社Workspaceの
   共有ドライブにサービスアカウントを「コンテンツ管理者」として追加。
3. **Cloudflare**：アカウント作成（無料）→ Pages / Workers / D1 プロジェクト作成。

## 8. 既存実装の扱い

- `bar-dashboard/`：画面・計算エンジンをPhase 1の移植元とする。移植完了までは
  デモ（localStorage版）として残す。
- `line-bot/`：スキーマ・計算エンジン・報告用テキスト生成を移植元とする。
  テキストパース（Webhook受信）系は新アプリでは使用しない。移植完了後にアーカイブ
  （削除またはREADMEに旧方式である旨を明記）する。
